# 文字 RPG 模拟器 - 框架设计 v1.0

**更新日期**: 2026-04-13  
**版本**: v1.0 (MVP 阶段)  
**目标**: 微信小程序上的多剧本文字 RPG 模拟器

---

## 一、产品定位

### 核心体验

用户在小程序中选择一个历史剧本（如虎牢关之战、商鞅变法、玄武门之变等），扮演一个角色，通过输入文字意图与 NPC 互动。LLM 作为"导演"，根据玩家意图和当前世界状态生成叙事（旁白 + 对话），并更新世界状态（HP、关系、声望等），形成互动循环。

### 目标用户

- 喜欢文字 MUD 游戏的用户
- 历史爱好者（三国/春秋战国/唐宋元明清）
- 喜欢剧情互动的用户

### 核心功能

1. 选择剧本（虎牢关/赤壁/商鞅变法/玄武门之变/...）
2. 选择扮演角色（剧本 NPC 或原创人物）
3. 选择互动 NPC
4. 输入文字意图
5. 观看叙事（旁白 + 对话）
6. 查看状态（HP、关系、声望、轮次）
7. 使用快捷选项（攻击/对话/投降/逃跑）

### 支持的剧本类型

**历史正剧**（公有领域）：
- 虎牢关之战（东汉末年）
- 商鞅变法（战国秦国）
- 玄武门之变（唐朝）
- 赤壁之战（三国）
- 靖康之变（北宋）
- 其他公有领域历史事件（1912 年之前）

**注意**：
- ❌ 不支持架空历史（版权风险）
- ❌ 不支持现代/当代剧本（版权风险）
- ✅ 只支持公有领域历史事件

## 三、核心模块

### 模块 1：剧本引擎（Engine）

**职责**：
- 加载剧本配置（从代码 import）
- **加载会话记忆**（从云数据库读取滚动摘要、关键事件、累计状态）
- 管理会话状态
- 构建 LLM prompt（注入时代背景、NPC 人设、关系状态、会话记忆）
- 调用 LLM
- 解析 LLM 输出
- 更新世界状态

**输入**：
- 用户 ID
- 剧本 ID
- 玩家意图

**输出**：
- 叙事文本（旁白 + 对话）
- 状态变化（HP、关系、声望等）
- 当前会话状态

**关键特性**：
- 剧本无关（不关心具体是哪个剧本）
- 状态管理（保存会话进度）
- **记忆加载**（从云数据库读取滚动摘要、关键事件、累计状态）
- LLM 封装（统一调用接口）
- 解析器（从叙事提取状态变化）
- **时代特征注入**（战国/三国/唐朝等不同时代的叙事风格）

**LLM 输出解析规则**：
- **输出格式要求**：
  - 必须包含旁白（narration）
  - 必须包含对话（dialogue）
  - **必须包含状态变化总结**（由 LLM 自主判断并输出）
- **LLM 自主判断状态变化**：
  - 在输出末尾添加 JSON 格式的状态变化总结
  - 由 LLM 根据叙事内容自主判断 HP 变化、关系变化
  - **完整输出示例**：
    ```json
    {
      "narration": "你突然拔出剑，寒光一闪！吕布瞳孔骤缩...",
      "dialogue": "吕布：'找死！'",
      "stateChanges": {
        "hp": -20,
        "relationship": -30,
        "reason": "玩家刺杀吕布，被方天画戟划伤"
      }
    }
    ```
- **校验规则**：
  - 检查是否包含旁白和对话
  - 检查状态变化总结是否为合法 JSON
  - 检查状态变化是否合理（如 HP 不能<0）
  - **如果 JSON 解析失败**：
    - 重试 1 次（让 LLM 重新生成）
    - 如果重试后仍失败，报错提示"生成失败，请重试"
    - ❌ **绝对不使用默认值**
- **错误提示文案规范**：
  - LLM 超时："响应超时，请重试"
  - 解析失败："生成失败，请重试"
  - 会话不存在："会话已过期，请重新开始"
  - 语气：友好、简洁、不带技术术语

**错误处理和降级**：
- **错误类型**：
  - LLM 超时（>300 秒无响应）
  - LLM 失败（API 报错）
  - 解析失败（无法提取状态变化）
  - 会话不存在（用户 ID 无效）
- **降级方案**：
  - LLM 超时：重试 1 次 → 切换备用模型（qwen3.5 → DeepSeek）→ 报错提示"响应超时，请重试"
  - LLM 失败：切换备用模型（qwen3.5 → DeepSeek）
  - 解析失败：重试 1 次 → 报错提示"生成失败，请重试"
  - 会话不存在：提示"会话已过期，请重新开始"
- **监控告警**：
  - 错误率> 5% 时告警
  - LLM 响应时间> 60 秒告警（1 分钟）
  - LLM 响应时间> 300 秒报错（5 分钟超时）
  - 解析失败率> 10% 告警

### 模块 2：剧本包（Scenario Pack）

**每个剧本包包含**：

- 剧本元信息（ID、名称、描述、版本、时代背景）
- NPC 定义（姓名、性格、动机、关系网、底线、时代特征）
- 事件定义（战斗、对话、投降、变法、政变等触发条件和效果）
- 地理定义（地点、连接、特殊规则、时代特征）
- 历史背景（开场旁白、世界观、时代特征）

**关键特性**：
- 独立打包（每个剧本自包含）
- 热插拔（新增剧本不用改引擎）
- 版本管理（可以迭代剧本内容）
- JSON 配置（非技术人员可编辑）
- **时代无关**（支持任意历史时期）

**时代特征示例**：
- 战国：变法、合纵连横、百家争鸣
- 三国：群雄割据、忠义、智谋
- 唐朝：政变、科举、开放
- 叙事风格：
  - 战国：古朴，称"君上"、"臣"
  - 三国：演义，称"某家"、"主公"
  - 唐朝：华丽，称"陛下"、"微臣"

**历史背景字段说明**：
- 开场旁白：剧本开场时的叙事（约 200 字）
- 世界观：基本设定（如"诸侯讨董"）
- 时代特征：时代关键词（如"忠义、智谋"）
- 示例：
  ```
  开场旁白："建安十三年的凛冬夜半..."
  世界观："诸侯讨董，虎牢关前对峙"
  时代特征："忠义、智谋、群雄割据"
  ```

**版本管理说明**：
- 版本号格式：`v{major}.{minor}`（如 v1.0）
- major 变更：剧本结构变化（需要用户重开）
- minor 变更：内容优化（用户可继续）
- 存储位置：剧本元信息中的 version 字段

**剧本包示例**：
- `scenario-hulaguan/` - 虎牢关之战（东汉末年）
- `scenario-shangyang/` - 商鞅变法（战国秦国）
- `scenario-xuanwumen/` - 玄武门之变（唐朝）
- `scenario-chibi/` - 赤壁之战（三国）

**NPC JSON 文件结构示例（lv-bu.json）**：

**事件 JSON 文件结构示例（combat.json）**：
```json
{
  "id": "combat",
  "name": "战斗事件",
  "type": "instant",
  "triggers": ["攻击", "刺杀", "挑战"],

    "hp": -20,

  }
```

**NPC 行为逻辑**：
- **被动响应**：玩家输入后，NPC 根据人设反应
- **主动触发**：达到条件时，NPC 主动发起事件
  - 例：关系< -80，吕布主动挑战
  - 例：关系> 60，吕布主动邀请加入
  - 例：HP< 20，NPC 主动求饶
- **触发条件**：
  - 关系阈值（如<-80 或>60）
  - HP 阈值（如<20）
  - 轮次阈值（如第 10 轮触发关键事件）
  - 地点触发（如进入某地触发剧情）

**事件系统细化**：
- **事件类型**：
  - 战斗事件：玩家攻击 NPC
  - 对话事件：玩家与 NPC 交谈
  - 投降事件：玩家投降
  - 逃跑事件：玩家逃跑
  - 特殊事件：变法、政变、结盟等（根据剧本类型）
- **事件结算**：
  - 即时结算：战斗（立即扣 HP）
  - 延迟结算：变法（多轮后生效）
  - 条件结算：结盟（达到条件后生效）
- **事件链**：
  - 前置事件：必须完成前置才能触发
  - 后置事件：触发后自动进入下一事件
  - 分支事件：根据选择进入不同分支
- **快捷选项触发逻辑**：
  - 攻击：触发战斗事件（HP-20，关系 -30）
  - 对话：触发对话事件（关系 +10）
  - 投降：触发投降事件（判定成功率）
  - 逃跑：触发逃跑事件（判定成功率）
  - 快捷选项不消耗轮次，但每轮只能使用一次
- **成功率计算逻辑**：
  - 投降成功率：基于关系值
    - 关系> 50: 90%
    - 关系 0-50: 50%
    - 关系<0: 10%
  - 逃跑成功率：基于 HP 和关系
    - HP> 80: 70%
    - HP 50-80: 50%
    - HP<50: 30%
    - 关系<-50: +20%（NPC 不在乎）
- **NPC 底线示例（吕布）**：
  - 被骂"三姓家奴"：立即愤怒，可能动手
  - 被刺杀：立即关押或处决
  - 被背叛：关系降至 -100，追杀

### 模块 3：世界初始化器（World Initializer）

**职责**：
- 创建纯净的世界状态副本
- 初始化 NPC 关系网
- 设置初始地点和事件
- 重置玩家状态

**工作流程**：
1. 用户选择剧本
2. 从剧本包读取原始配置
3. 深拷贝一份到会话存储
4. 设置初始关系值
5. 放置玩家到起始地点
6. 返回纯净世界状态

**关键特性**：
- 每次新开剧本都是全新生成
- 不影响剧本包原始配置
- 支持多用户同时开同一个剧本
- 会话结束后自动清理

**玩家角色创建流程**：
1. 用户选择剧本
2. 选择角色类型（剧本 NPC 或原创人物）
3. 如果选剧本 NPC：从剧本包读取角色属性
4. 如果选原创人物：用户输入名称和描述，引擎生成默认属性
5. 深拷贝到会话数据

**原创人物默认属性**：
- HP：100
- maxHp：100
- 性格：根据用户描述生成（如"勇敢"→易怒）
- 初始关系：所有 NPC 为 0（陌生）

### 模块 4：记忆持久化（Memory Persistence）

**职责**：
- 每轮对话后，将叙事结果追加到会话存储
- 保证后续推理不丢失记忆
- **新开剧本时清理旧会话**（无定期清理）

**数据存储策略**：
- **剧本包**：JSON 文件，代码中 import 读取（只读，不修改）
- **会话数据**：云数据库存储（可写，深拷贝副本）

**工作流程**：
1. 用户输入意图
2. 引擎加载会话数据（云数据库）
3. LLM 生成叙事 + 状态变化 JSON
4. 解析 JSON，更新会话数据
5. 保存到云数据库（持久化）
6. 返回结果给用户

**关键特性**：
- ✅ 每轮追加（不覆盖剧本包）
- ✅ 会话隔离（每个用户独立）
- ✅ **单会话制**（用户只能有一个活跃会话）
- ✅ **新开即清空**（新开会话时自动删除旧会话）
- ✅ **副本存储化**（深拷贝剧本包到云数据库）
- ✅ 新开剧本时清理（节省存储）

**会话 ID 生成规则**：
- 格式：`{userId}_{scenarioId}`
- 示例：`user123_hulaguan`
- 目的：保证单会话制（每个用户每个剧本只有一个会话）

**云数据库表结构**：
- 表名：`sessions`
- 字段：
  - `_id`: String (主键)
  - `sessionId`: String (索引)
  - `userId`: String (索引)
  - `scenarioId`: String
  - `npcs`: Object
  - `relationships`: Object
  - `player`: Object
  - `recentSummary`: String
  - `keyEvents`: Array
  - `history`: Array
  - `createdAt`: Date
  - `updatedAt`: Date

**引擎处理流程（伪代码）**：
```javascript
async function process(session, intent) {
  // 1. 加载记忆
  const memory = {
    recentSummary: session.recentSummary,
    keyEvents: session.keyEvents,
    cumulativeState: session.cumulativeState
  };
  
  // 2. 构建 prompt
  const prompt = buildPrompt(session.npcs, memory, intent);
  
  // 3. 调用 LLM
  const response = await callLLM(prompt);
  
  // 4. 解析输出
  const changes = parseResponse(response);
  
  // 5. 更新状态
  session.player.hp += changes.hp;
  session.relationships.current += changes.relationship;
  
  // 6. 更新记忆
  updateMemory(session, intent, response, changes);
  
  // 7. 返回结果
  return {
    narration: response.narration,
    dialogue: response.dialogue,
    changes: changes,
    state: session
  };
}
```

**Prompt 构建逻辑（伪代码）**：
```javascript
function buildPrompt(npcs, memory, intent) {
  return `
【NPC 人设】
${npcs.current.name}（${npcs.current.personality}）

【关系状态】
玩家→${npcs.current.name}：${getRelationshipLabel(npcs.current.relationship)}（${npcs.current.relationship}）

【关键事件】
${memory.keyEvents.map(e => `R${e.round}: ${e.event} → ${e.impact}`).join('\n')}

【近况】
${memory.recentSummary}

【累计状态】
HP: ${memory.cumulativeState.hp}/${memory.cumulativeState.maxHp}
轮次：${memory.cumulativeState.totalRounds}

【意图】
${intent}

请生成叙事（旁白 + 对话 + 状态变化 JSON）：
`;
}
```

**记忆更新逻辑（伪代码）**：
```javascript
function updateMemory(session, intent, response, changes) {
  // 1. 更新滚动摘要
  const summary = `R${session.currentRound}: ${intent.substring(0, 20)}...`;
  session.recentSummary.push(summary);
  if (session.recentSummary.length > 3) {
    session.recentSummary.shift();  // 丢弃最早的
  }
  
  // 2. 检测关键事件
  if (isKeyEvent(changes)) {
    session.keyEvents.push({
      round: session.currentRound,
      event: intent,
      impact: getImpactDescription(changes)
    });
  }
  
  // 3. 更新累计状态
  session.cumulativeState.totalRounds++;
  session.cumulativeState.hp = session.player.hp;
}

function isKeyEvent(changes) {
  return Math.abs(changes.relationship) > 50 || 
         changes.hp < -30 || 
         ['投降', '逃跑', '刺杀'].some(k => intent.includes(k));
}
```

**关键事件 impact 字段说明**：
- 作用：记录该事件对后续剧情的影响
- 用途：注入 prompt，让 LLM 知道之前的事件后果
- 示例：
  - "吕布好奇"→后续可能主动询问奇物
  - "关系降至 -80"→后续可能主动挑战

**滚动摘要生成逻辑**：
- 每轮结束后更新
- 格式：`R{轮次}: {意图简短摘要}`
- 示例：`R18: 认输被关；R17: 绝食被冷笑；R16: 炸长江被警惕`
- 长度限制：不超过 300 token
- 超出时丢弃最早的轮次

**累计状态字段定义**：
- totalRounds: 总轮次
- ending: 已解锁结局（数组）
- status: 当前状态（自由/被关押/逃跑中）

**关键事件检测规则**：
- 关系阈值突破：|关系变化 | > 50
- HP 阈值突破：HP 变化 < -30 或 > 30
- 特殊行为：投降/逃跑/刺杀/结盟/展示奇物
- 轮次阈值：每 10 轮自动标记

```json
  "relationships": { "lv-bu": -60 },
  "player": { "hp": 80, "maxHp": 100 },
  
  // 滚动摘要（最近 3 轮，用于 prompt）
  "recentSummary": "R18 认输被关；R17 绝食被冷笑；R16 炸长江被警惕",
  
  // 关键事件标记（永久保存，用于记忆）
  "keyEvents": [
    { "round": 5, "event": "展示手机", "impact": "吕布好奇" },
    { "round": 10, "event": "辱骂三姓家奴", "impact": "关系降至 -80" }
  ],
  
  // 完整历史（用于存档，不注入 prompt）
  "history": [
    {
      "round": 1,
      "intent": "我要刺杀吕布",
      "narration": "你突然拔出剑...",
      "changes": { "hp": -20, "relationship": -30 },
      "timestamp": "2026-04-14T12:05:00Z"
    }
  ]
}
```

**Prompt 注入策略**：
- ✅ 从会话数据读取（云数据库）
- ✅ 滚动摘要（最近 3 轮，约 300 token）
- ✅ 关键事件标记（永久保存，约 200 token）
- ✅ 累计状态（约 100 token）
- ✅ 合计：约 600 token/轮（安全）

### 模块 5：会话管理（Session）

**职责**：
- 创建会话（选择剧本）
- 保存会话进度
- **新开剧本时清理旧会话**（无定期清理）

**会话数据**：
- 用户 ID
- 当前剧本 ID
- 当前互动 NPC
- 当前轮次
- **玩家角色信息**：
  - 角色类型：剧本 NPC 或原创人物
  - 角色名称
  - 角色属性（HP、maxHp、性格等）
- **NPC 副本数据**（从剧本包深拷贝）
- **状态类型**：
  - 基础状态：HP、maxHp
  - 关系状态：与当前 NPC 的关系值（-100 到 100）
  - 位置状态：当前地点 ID
  - 时间状态：当前轮次
  - 自由状态：自由/被关押/逃跑中
  - 累计统计：总轮次、结局、成就
- 滚动摘要（最近 3 轮，用于 prompt）
- 关键事件标记（永久保存，用于记忆）
- 累计统计（总轮次、结局）

**关键特性**：
- **云数据库存储**（持久化，支持继续体验）
- **单会话制**（用户只能有一个活跃会话）
- **新开即清空**（新开会话时自动删除旧会话）
- **副本存储化**（深拷贝剧本包到云数据库）
- 状态压缩（只存必要数据）

**初始值定义**：
- HP：100（满血）
- maxHp：100
- 关系：0（陌生）
- 关系范围：-100（死敌）到 100（生死之交）

**累计统计字段**：
- totalRounds: 总轮次
- ending: 已解锁结局（数组）
- status: 当前状态（自由/被关押/逃跑中）

### 模块 6：LLM 适配器（LLM Adapter）

**职责**：
- 统一 LLM 调用接口
- 支持切换模型
- Token 计数和管理

**支持的模型**：
- qwen3.5-plus（当前主力）
- DeepSeek（备选，成本更低）
- MiniMax（语音合成）

**关键特性**：
- 模型无关（随时切换）
- Token 管理（防止超限）
- 错误处理（重试机制）

**性能优化策略**：
- **响应时间优化**：
  - 云函数就近部署（选择离用户最近的区域）
  - LLM 调用并行化（同时调用多个模型，取最快的）
  - 缓存常用 prompt 模板（减少构建时间）
- **成本优化**：
  - Token 压缩（滚动摘要、状态压缩）
  - 模型切换（简单意图用 DeepSeek，复杂意图用 qwen3.5）
  - 会话上限（单会话 50 轮，超限提示重开）
- **并发优化**：
  - 云函数自动扩缩容
  - 会话隔离（每个用户独立会话，互不影响）
  - 限流策略（单用户每秒最多 1 次请求）

## 四、数据流（完整流程）

### 初次打开剧本

1. 用户打开小程序
2. 选择剧本（如虎牢关）
3. 选择角色类型（剧本 NPC 或原创人物）
4. 世界初始化器创建纯净世界状态
5. 创建会话（记录剧本 ID、初始状态）
6. 输入意图（如"我要刺杀吕布"）

### 后续对话

6. 云函数调用引擎
7. 引擎加载会话状态（内存）
8. **加载会话记忆**（云数据库：滚动摘要 + 关键事件 + 累计状态）
9. 构建 prompt（NPC 人设 + 关系 + 近况 + 记忆 + 意图）
10. 调用 LLM
11. 解析 LLM 输出（提取 HP 变化、关系变化）
12. 更新会话状态（内存）
13. 追加到记忆持久化（云数据库）← 新增
14. 返回结果（叙事 + 变化 + 状态）
15. 小程序展示叙事和状态

### 重开剧本

- 删除旧会话
- 重新执行步骤 1-4（全新生成）

## 五、关键设计决策

### 决策 1：剧本包独立 vs 内置

**选择**：独立剧本包

**理由**：
- 新增剧本不用改代码
- 可以外包剧本创作
- 支持付费剧本包

---

### 决策 2：状态存储方式

**选择**：内存（会话状态）+ 云数据库（持久化）

**理由**：
- 读写快（毫秒级）
- 支持继续体验（云数据库持久化）
- 成本低（云函数 + 云数据库免费额度）

---

### 决策 3：LLM 调用方式

**选择**：直接 API 调用（不微调）

**理由**：
- 零训练成本
- 随时切换模型
- prompt 优化即可

---

### 决策 4：上下文管理策略

**选择**：滚动摘要 + 状态压缩

**理由**：
- 固定 token 消耗（约 500/轮）
- 支持 50+ 轮对话
- 实现简单

**具体策略**：
- 只保存最近 3 轮摘要（不保存完整历史）
- 只保存当前互动 NPC 的关系（不保存完整关系网）
- 关键事件标记（永久保存，用于记忆）
- NPC 人设模板化（不每轮都传完整人设）

---

### 决策 5：玩家角色类型

**选择**：支持剧本 NPC 和原创人物两种

**理由**：
- 剧本 NPC：用户可以直接扮演吕布、曹操等知名角色，体验历史
- 原创人物：用户可以创建自定义角色，自由发挥
- 两种类型各有受众，增加游戏可玩性

**实现方式**：
- 剧本 NPC：从剧本包读取角色属性（HP、性格、关系网等）
- 原创人物：用户提供名称和简单描述，引擎生成默认属性

---

### 决策 6：会话存储策略

**选择**：副本存储化（深拷贝剧本包到云数据库）

**理由**：
- 简单直观（深拷贝逻辑清晰）
- 支持继续体验（云数据库持久化）
- 支持重开（删除 + 深拷贝）
- 读取快（剧本包代码读取）
- 成本低（剧本包无需数据库）

**实现方式**：
- 剧本包：JSON 文件，代码中 import 读取（只读）
- 会话数据：云数据库存储（可写，深拷贝副本）
- Prompt 注入：从会话数据读取（不直接读取剧本包）

---

### 决策 7：Prompt 注入策略

**选择**：滚动摘要 + 关键事件标记 + 累计状态

**理由**：
- 避免上下文撑爆（只注入约 600 token/轮）
- 避免记忆丢失（关键事件永久保存）
- 支持长期对话（50+ 轮不撑爆）

**注入内容**：
- NPC 人设（模板化，约 30 token）
- 关系状态（精简版，约 50 token）
- 关键事件标记（永久保存，约 200 token）
- 累计状态（约 100 token）
- 滚动摘要（最近 3 轮，约 300 token）
- 玩家意图（约 50 token）
- 系统指令（约 100 token）
- **合计：约 630 token/轮**（安全）

## 六、Token 消耗预估

### 单轮 prompt 组成

- NPC 人设（模板化）：约 30 token
- 关系状态（精简版）：约 50 token
- 历史摘要（最近 3 轮）：约 300 token
- 关键事件标记（永久保存）：约 200 token
- 累计状态：约 100 token
- 玩家意图：约 50 token
- 系统指令：约 100 token

**合计**：约 630 token/轮（安全）

### 对话轮次支持

- 20 轮对话：约 11600 token（安全）
- 50 轮对话：约 29000 token（安全，qwen3.5 支持 32k）
- 100 轮对话：约 58000 token（超出，需要清理会话）

**建议**：单会话上限 50 轮，达到后提示"开启新剧情"
