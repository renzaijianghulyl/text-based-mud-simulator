# 仓库治理与发布模型

## 单一事实来源

本仓库以 **`main`（或团队约定的默认主干分支）为唯一事实来源**：叙事引擎、CLI、剧本包、微信云函数与小程序前端的演进都在同一棵提交历史上完成。

**不推荐**长期维护两条平行分支、各自手写同一套引擎逻辑（例如 `main` 与 `opensource` 各改一遍 `src/engine`）。那样会很快产生合并冲突与线上/开源行为不一致。

## 若需「公开树更瘦」或独立公开仓

从主干**自动化导出**（CI、脚本或 `git filter-repo` / subtree）到公开镜像仓库或 `publish/engine-only` 分支，合并方向固定为：

**私有/集成仓库 → 公开导出**

公开侧原则上只接受文档与小范围适配，避免在公开仓单独堆功能后再反向合并进主干。

## 小程序发版

小程序发版使用 **tag**（例如 `weapp-v0.3.0`）或 `release/weapp-*` 分支指向主干上已验证的提交即可，无需为「开源」单独维护第二套产品分支。
